## Overview

Supasheet includes pre-built migrations in `/supabase/migrations/` that create all necessary schemas, tables, and functions. You just need to apply them.

## Local Setup (Development)

<Steps>
<Step>
### Start Supabase

```bash
npx supabase start
```

Starts local Supabase instance with PostgreSQL.
</Step>

<Step>
### Apply Migrations

```bash
npx supabase db reset
```

Applies all migrations from `/supabase/migrations/`.
</Step>

<Step>
### Verify Setup

```bash
npx supabase status
```

Check that all services are running.
</Step>
</Steps>

## Cloud Setup (Production)

<Steps>
<Step>
### Create Supabase Project

1. Go to [supabase.com/dashboard](https://supabase.com/dashboard)
2. Click "New Project"
3. Choose organization and region
4. Set database password
5. Wait for project creation (~2 minutes)
</Step>

<Step>
### Link Project

```bash
npx supabase link --project-ref your-project-ref
```

Get `project-ref` from dashboard URL: `https://supabase.com/dashboard/project/[project-ref]`
</Step>

<Step>
### Push Migrations

```bash
npx supabase db push
```

Applies all local migrations to cloud database.
</Step>
</Steps>

## What Gets Created

The migrations in `supabase/migrations/` build the complete platform layer:

- **Schemas**: `supasheet` (Supasheet's meta + RBAC + audit/notifications/comments), plus `public` (yours to use)
- **Meta layer**: `supasheet.tables`, `columns`, `views`, `materialized_views` — materialized views refreshed manually via `select supasheet.refresh_metadata();` after DDL changes (see [Database Schema](/database-schema#refreshing-the-meta-layer))
- **Users**: `supasheet.users` with triggers to mirror `auth.users`
- **RBAC**: native Postgres roles (`x-admin`/`admin`/`user`, created via `CREATE ROLE`), a Custom Access Token Hook wiring `auth.users.raw_app_meta_data->>'role'` into the JWT, and the helper `has_role()` — access to resources is granted directly via `GRANT`/`REVOKE`, not a permissions table
- **Audit log**: `supasheet.audit_logs` and `audit_trigger_function()`
- **Notifications**: `supasheet.notifications` + `user_notifications` + `create_notification()` / resolver helpers
- **Comments**: `supasheet.comments` and `get_comments()`
- **Templates**: `get_templates()` and `apply_template()`
- **Custom types**: `FILE_OBJECT`, `FILE`, `EMAIL`, `TEL`, `RATING`, `PERCENTAGE`, `URL`, `DURATION`, `COLOR`, `AVATAR`, `RICH_TEXT`
- **Storage buckets**: `public`, `personal`, `uploads` (also used for account avatars, under `uploads/auth/{uid}/...`) with RLS policies
- **Discovery functions**: `get_schemas()`, `get_tables()`, `get_views()`, `get_columns()`, `get_charts()`, `get_widgets()`, `get_reports()`, `get_templates()`

## Schema Exposure (Required)

After applying migrations, you **must** expose the `supasheet` schema via the PostgREST API. The `public` schema is already exposed by default.

<Steps>
<Step>
### Edit Supabase Configuration

Add the `supasheet` schema to `/supabase/config.toml`:

```toml
[api]
schemas = ["public", "supasheet"]
```

This exposes:
- `public` - Your application tables (exposed by default)
- `supasheet` - All Supasheet internal tables including accounts, roles, permissions, dashboards, charts, and reports
</Step>

<Step>
### Restart Supabase (Local)

For local development, restart Supabase to apply the configuration:

```bash
npx supabase stop
npx supabase start
```
</Step>

<Step>
### Update Cloud Configuration (Production)

For cloud/production, update via the Supabase Dashboard:

1. Go to **Project Settings** → **Data API**
2. Find **Exposed schemas** section
3. Add `supasheet` to the list
4. Save changes
</Step>
</Steps>

<Callout type="warning">
**Important**: Without exposing the `supasheet` schema, Supasheet won't be able to access authentication, authorization, configuration, dashboards, charts, and reports. The application will not function correctly.
</Callout>

## Adding Your Own Tables

Create new migrations for your application tables:

```bash
npx supabase migration new add_products_table
```

Each migration that adds tables typically does five things — create the table, revoke default grants then grant exactly what each role needs, enable RLS + policies, (optionally) attach the audit trigger, and refresh the meta layer:

```sql
CREATE TABLE store.products (...);

REVOKE ALL ON TABLE store.products FROM authenticated, service_role;
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE store.products TO "user";

ALTER TABLE store.products ENABLE ROW LEVEL SECURITY;
-- ... policies, audit trigger ...

-- Required last step: supasheet.tables/columns/views are materialized
-- views and won't pick up your new table until you refresh them.
select supasheet.refresh_metadata();
```

<Callout type="warn">
There are no event triggers watching for schema changes — `select supasheet.refresh_metadata();` is the only thing that syncs the meta layer. Forgetting it is the most common reason a new table doesn't show up in the UI.
</Callout>

After running migrations, regenerate types:

```bash
npx supabase gen types typescript --local \
  --schema public --schema supasheet --schema store \
  > src/lib/database.types.ts
```
